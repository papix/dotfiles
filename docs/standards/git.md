# Git使用規約

## コミット規約

### メッセージフォーマット
```
<type>: <subject>

[optional body]
```

### タイプ
- `feat`: 新機能の追加
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの意味に影響しない変更（空白、フォーマット等）
- `refactor`: バグ修正でも機能追加でもないコード変更
- `perf`: パフォーマンス改善
- `test`: テストの追加・修正
- `chore`: ビルドプロセスやツールの変更

### 例
```bash
feat: tmuxセッション自動起動機能を実装

- ghqディレクトリに基づいてセッション名を生成
- VSCode/Cursor環境では無効化
- DISABLE_AUTO_TMUX環境変数で制御可能
```

## ブランチ戦略

### ブランチ名
- `main`: プロダクションブランチ
- `feature/<feature-name>`: 新機能開発
- `fix/<issue-number>`: バグ修正
- `refactor/<target>`: リファクタリング

### フロー
1. mainから新しいブランチを作成
2. 機能を実装
3. mainにPRを作成
4. レビュー後マージ

## コミット粒度

### 推奨
- 論理的な単位でコミット
- 1つのコミットで1つの変更
- ビルドが通る状態を保つ

### 避けるべき
- 巨大なコミット
- 複数の無関係な変更を含むコミット
- WIPコミット（完了時にsquash）

## ファイル管理

### .gitignore
```
# ローカル設定
*.local
.zshrc.local

# エディタ設定
.vscode/
.idea/

# OS固有
.DS_Store
Thumbs.db

# ビルド成果物
*.zwc
```

### 機密情報
- APIキーやパスワードは絶対にコミットしない
- 環境変数や別ファイルで管理
- 誤ってコミットした場合は履歴から完全に削除
